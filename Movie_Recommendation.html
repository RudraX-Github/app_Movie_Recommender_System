Core Technologies & Methods

Libraries & Frameworks

  • Streamlit: The core Python framework for the interactive web app frontend.
  • Pandas: Used for all data manipulation and CSV processing in the backend.
  • Scikit-learn: The heart of the ML model, used for CountVectorizer and cosine_similarity.
  • Numpy: For numerical operations and optimizing the similarity matrix size.

Core ML Methods

  • Content-Based Filtering: The overall recommendation strategy. It suggests movies based on their shared attributes (tags).
  • Bag-of-Words: The CountVectorizer model, which converts text tags into a numerical vector matrix.
  • Cosine Similarity: The algorithm used to calculate the "similarity" score (from 0 to 1) between any two movies.
  • Feature Engineering: The critical process of creating the tags column from raw text.

App Techniques

  • State Management: Using st.session_state to control the UI, especially for the popup modal.
  • Custom CSS Injection: Using st.markdown(unsafe_allow_html=True) to apply custom styles, fonts, and 3D effects.
  • Data Caching: Using @st.cache_data to load the large .pkl files only once, making the app fast.
  • Robust Error Handling: Architecting the app to cleanly handle data loading failures.

The Development Journey

V0.x: The Blueprint

The Data Science Backend

  • Merge & Clean: Transformed raw TMDB 5000 CSVs into a single, clean Pandas DataFrame.
  • Feature Engineering: Created the critical tags column by combining overview, cast, crew, and genres.
  • Model & Export: Used CountVectorizer and cosine_similarity to build and export movies_dict.pkl and similarity.pkl.
V1.0: The First Spark

Basic Proof of Concept

The first Streamlit app was born. It used @st.cache_data to load the models and featured a simple st.selectbox and st.button. It proved the core logic: a user could pick a movie and see a text-based list of 5 recommendations.

V2.0: The "Expert" Upgrade

The Visual & UI Polish

Functionality was just the start. We created fetch_poster to call the TMDB API, upgrading the UI from text to a 5-column grid of posters. A large custom CSS block was injected, adding a pro font, full dark mode, and 3D "tilt-on-hover" cards.

[Crisis] V3.0: The First Crisis

The Network Block

Disaster. The app's core visual feature, fetching posters, suddenly failed with a ConnectionResetError. The Python script was being blocked from api.themoviedb.org. Even switching to API v4 Auth failed, proving it was a network block. The app was downgraded to a text-only list.

[Solution] V4.0: The Architectural Pivot

The Data Pipeline Solution

The solution wasn't to fix the connection, but to bypass it. We modified the backend notebook to save the poster_path in movies_dict.pkl. The Streamlit app no longer used requests. Instead, it built the full image URL and passed it to st.image, forcing the user's browser (not Python) to fetch the image.

V5.0: Rich Features & Refinement

Polishing the UX

  • Bug Fix: A "You Selected" card revealed spaceless names (e.g., SamWorthington). Fixed with a regex helper in load_data().
  • Feature: Upgraded recommend to return Top 8 similar and 2 "Wildcard" (least similar) movies, split into st.tabs for a cleaner UX.
[Crisis] V7.0: The Environment Crash

The Broken Modal

A new "floating popup" for details crashed the app with a TypeError on st.dialog(). The deployment environment's Streamlit version was too old. The fix was a "Hybrid Rendering" hack: building a custom modal by injecting HTML <div> tags with st.markdown around normal Streamlit components.

[Solution] V8.0: Production Hardening

The "Zombie Script" Fix

The final boss: a "Zombie Script" error. On deployment, if load_data() failed (e.g., a URL typo), the script kept running past Sst.stop(), crashing with a NameError. The fix: We refactored the entry point so the UI-building function (main_app()) is only called if load_data() completes successfully. This is the key to a production-ready app.

Major Challenges & Resolutions

The API Network Block

A ConnectionResetError completely blocked the Python script from fetching posters, killing the app's visual appeal.

Resolution:

Re-architected the data pipeline. We saved the poster_path in the backend .pkl file, then passed the full image URL to st.image. This forced the user's browser to fetch the image, bypassing Python's network block entirely.

The Streamlit Version Crash

A TypeError on st.dialog() crashed the app. The deployment environment's Streamlit version was too old, and no built-in modal component worked.

Resolution:

Built a custom modal from scratch using "Hybrid Rendering." We used st.markdown to inject HTML <div> tags (styled with backdrop-filter) around standard Streamlit components to create a functional, reliable modal.

The Deployment "Zombie Script"

On deployment, a data loading failure (e.g., a URL typo) would cause st.stop(), but the script would keep running, crashing with a NameError as it tried to access variables that were never created.

Resolution:

Refactored the app's entry point. All UI code was moved into a main_app() function, which is only called if load_data() completes successfully. This ensures the app either loads fully or stops cleanly.
